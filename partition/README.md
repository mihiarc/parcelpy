# Parcel Partition Module

A Python-based module for batch processing land parcel data to prepare for ingestion into Google BigQuery.

## Key Features

- Converts GDB or GeoPackage files to GeoParquet format
- Processes parcels county by county (can use parallel processing with multiple workers)
- Validates and repairs geometries for each county
- Identifies and fixes overlapping parcel geometries
- Produces clean, county-level parcel files in GeoParquet format
- Handles memory constraints by loading and processing one county at a time
- Provides detailed logging and progress tracking

## Overlap Correction Algorithm

The parcel overlap correction algorithm addresses an important challenge in geospatial data: parcel polygons that incorrectly overlap with neighboring parcels.

### How It Works

1. **Spatial Indexing**: The algorithm efficiently identifies all overlapping parcel pairs using spatial indexing.

2. **Overlap Distribution**: Rather than removing overlaps (which would reduce total area), the algorithm redistributes the overlapping area between the two parcels. This maintains the total land area while ensuring each piece of land is assigned to exactly one parcel.

3. **Boundary Correction**: The algorithm uses either:
   - A Voronoi diagram based on parcel centroids to determine the most logical boundary
   - Or a simpler method splitting along the line between centroids if Voronoi fails

### Purpose and Benefits

- **Maintains Parcel Completeness**: Every part of the original parcels is preserved, just redistributed to eliminate overlaps.
- **Creates Seamless Boundaries**: Parcels share clean, non-overlapping boundaries after correction.
- **Preserves Total Land Area**: The total claimed area remains unchanged before and after correction.
- **Eliminates Double-Counting**: After correction, each piece of land is claimed by exactly one parcel, preventing double-counting in subsequent analysis.

### Metrics to Evaluate Correction

The algorithm tracks several key metrics to quantify the magnitude of the overlap correction:

- **Total Overlap Area**: The sum of all overlap areas, showing how much double-counting existed before correction.
- **Overlap Percentage**: What percentage of the total claimed area was overlapping.
- **Average Overlap Size**: The mean size of overlaps, helping characterize the nature of the overlap problem.
- **Parcels with Overlaps**: An estimate of how many parcels were involved in overlaps, showing how widespread the issue was.

These metrics are included in the summary reports generated by the pipeline.

## Components

- **config.py**: Central configuration module
- **config_manager.py**: Manages access to configuration settings
- **main.py**: Main entry point for the pipeline
- **convert_gdb_to_geoparquet.py**: Converts state-level GDB to GeoParquet format
- **split_and_fix_parcels.py**: Splits data by county and fixes parcel overlaps
- **county_manager.py**: Manages county code data and county-related operations
- **crs_manager.py**: Handles coordinate reference system operations and transformations
- **processing_manager.py**: Centralizes parallel processing functionality 
- **geometry_engine.py**: Centralizes geometry validation, overlap detection, and resolution
- **io_manager.py**: Provides unified I/O operations for geospatial data
- **log_manager.py**: Centralizes logging functionality
- **batch_run_states.sh**: Shell script for batch processing multiple states

## Architecture

The system follows a modular architecture with several key components:

1. **Configuration**: The `config.py` and `config_manager.py` modules provide centralized configuration for the entire system, including file paths, directory structures, and processing parameters.

2. **County Management**: The `county_manager.py` module handles all county-related operations, including:
   - Loading county FIPS codes and names
   - Extracting county metadata
   - Mapping between different county ID formats
   - Calculating county areas
   - Filtering data by county boundaries

3. **CRS Management**: The `crs_manager.py` module handles all coordinate reference system operations:
   - Determining if a CRS is suitable for area calculations
   - Reprojecting data for accurate area measurements
   - Standardizing CRS handling across the codebase
   - Validating and fixing geometry after projection changes

4. **Geometry Processing**: The `geometry_engine.py` module centralizes all geometry operations:
   - Validation and repair of invalid geometries
   - Spatial index-based overlap detection
   - Resolution of overlapping parcel boundaries
   - Area calculation and statistical analysis
   - Complete processing pipeline for parcel datasets

5. **I/O Operations**: The `io_manager.py` module centralizes all file operations:
   - Reading/writing multiple geospatial formats
   - Filtered loading to read only relevant data
   - Automatic format detection and appropriate handling
   - Standardized error handling and path resolution

6. **Parallel Processing**: The `processing_manager.py` module centralizes all parallel processing:
   - Managing worker pools for concurrent processing
   - Optimizing memory usage by assigning one county per worker
   - Providing consistent interfaces for parallelization
   - Handling graceful shutdown and error recovery
   - Monitoring system resources during processing

7. **Logging**: The `log_manager.py` module centralizes all logging:
   - Consistent logging interface across all modules
   - Colored console output and file logging
   - Progress tracking and reporting
   - Integration with shell scripts

8. **Pipeline Orchestration**: The `main.py` module provides the main entry point for the pipeline, handling argument parsing, logging setup, and coordinating the processing steps.

9. **Batch Processing**: The `batch_run_states.sh` script enables batch processing of multiple states sequentially or in parallel.

## Processing Flow

1. **GDB to GeoParquet Conversion**: Convert state-level GDB files to GeoParquet format.
2. **County-Level Processing**: For each county in the state:
   - Load only that county's data using filtering from the source GeoParquet
   - Validate and repair geometries using the geometry engine
   - Identify overlapping parcels using spatial indexing
   - Fix overlaps using configurable resolution algorithms
   - Calculate and report area statistics
   - Save clean county-level parcel file

## System Requirements

- Python 3.8 or higher
- GeoPandas and dependencies (GDAL, Fiona, Shapely, PyProj)
- Pandas, NumPy, and other scientific Python libraries
- Sufficient RAM to process the largest county in a state (8GB recommended)

## Geometry Engine

The Geometry Engine module (`geometry_engine.py`) provides the core functionality for processing parcel geometries:

### Key Features

1. **Geometry Validation and Repair**
   - Validates all input geometries
   - Repairs invalid geometries using Shapely's `make_valid` function
   - Ensures all geometries are of the correct type (Polygon/MultiPolygon)
   - Tracks validation statistics and errors

2. **Overlap Detection with Spatial Indexing**
   - Uses spatial indexing for efficient identification of potentially overlapping parcels
   - Handles edge cases and errors during intersection operations
   - Tracks and reports on all detected overlaps

3. **Overlap Resolution Algorithm**
   - Implements a configurable method for resolving geometric overlaps
   - Supports different split strategies:
     - Equal distribution of overlap (50/50)
     - Custom ratio distribution (e.g., 70/30)
     - Full assignment to one parcel (0/100 or 100/0)
   - Ensures resulting geometries are valid after modification

4. **Area Calculations and Statistics**
   - Calculates comprehensive area statistics for parcel datasets
   - Integrates with the CRS Manager to ensure proper projections for area calculations
   - Converts between square meters and acres
   - Tracks statistical information about overlaps and resolutions

### Usage

The geometry engine can be used directly or through the `split_and_fix_parcels.py` module:

```python
from geometry_engine import geometry_engine
import geopandas as gpd

# Load parcel data
parcels = gpd.read_file("parcels.shp")

# Process parcels using the geometry engine
fixed_parcels, stats = geometry_engine.process_parcel_data(parcels)

# Access statistics
print(f"Found {stats['overlaps']['total_overlaps']} overlaps")
print(f"Fixed {stats['overlaps']['fixed_overlaps']} overlaps")
print(f"Total overlap area: {stats['overlaps']['total_overlap_area_acres']:.2f} acres")
```

## Parallel Processing

The system utilizes parallel processing for improved performance:

- **Worker-Based Parallelism**: Using the `processing_manager.py` module, the system can process multiple counties in parallel
- **Configurable Worker Count**: Control the number of parallel workers via command-line arguments
- **Memory Management**: Each worker processes one county at a time, loading only that county's data
- **Task Distribution**: Processing tasks are evenly distributed across available CPU cores

## Command-Line Usage

```bash
# Process a state with default settings
python main.py --state MT

# Process with custom options
python main.py --state MT --workers 4

# Run a specific step in the pipeline
python main.py --state MT --step convert  # Only run GDB conversion
python main.py --state MT --step split    # Only run county splitting and fixing
python main.py --state MT --step validate # Only run validation and stats

# Process a specific county
python main.py --state MT --county-name "Yellowstone" 
python main.py --state MT --county-code "30111"

# Batch process multiple states
./batch_partition_states.sh # default workers is 4
```

## Command-Line Arguments

- `--state, -s`: State abbreviation (e.g., MT, ND, SD)
- `--step, -t`: Which step to run (all, convert, split, validate)
- `--gdb-file`: Custom path to GDB file (optional)
- `--input-file, -i`: Custom path to input file for splitting
- `--output-dir, -o`: Custom path to output directory
- `--county-code`: Process only a specific county FIPS code
- `--county-name`: Process only a specific county name
- `--county-column`: Column containing county FIPS code (default: COUNTYFP)
- `--workers, -w`: Number of parallel workers for processing
- `--no-chunking`: Disable chunking during GDB conversion
- `--chunk-size`: Chunk size for GDB conversion (default: 100000)
- `--continue-on-error, -c`: Continue execution if a step fails

## Batch Fix Overlap Workflow

This module processes parcel data from state geodatabases, converting them to GeoParquet format, splitting by county, and fixing overlapping geometries. The workflow consists of these main components:

### 1. Setup Phase
- `setup_county_data.py`: Extracts county data from TIGER/Line shapefiles, creating a JSON file with county information (FIPS codes, names, areas) for all states.

### 2. Main Processing Pipeline (main.py)
The main script coordinates the complete processing workflow with these key steps:
- Ensures required directories exist
- Loads county data for the specified state
- Executes three main processing steps:
  - Step 1: Converts GDB to GeoParquet format
  - Step 2: Splits data by county and fixes parcel overlaps
  - Step 3: Validates results and generates statistics

### 3. GDB to GeoParquet Conversion (convert_gdb_to_geoparquet.py)
- Lists and processes all layers in the geodatabase
- Handles large datasets by processing in configurable chunks to manage memory usage
- Converts and saves data in GeoParquet format with proper projections

### 4. County Processing and Overlap Fixing (split_and_fix_parcels.py and geometry_engine.py)
- Processes parcels county by county (can use parallel processing with multiple workers)
- Loads only the specific county's data using filtering
- Uses the geometry engine to validate, detect and fix overlapping geometries
- Applies configurable overlay resolution algorithms
- Generates comprehensive statistics on the processing results

### 5. Batch Processing (batch_run_states.sh)
- Automates processing multiple states sequentially
- Sets up county data if not already done
- Processes each state's GDB file with optimized parameters
- Organizes output into state-specific directories
- Provides comprehensive logging and error handling
- Supports processing a single state with `-s STATE_CODE` option

The system includes robust features for:
- Parallel processing with configurable worker count
- Memory management by processing one county per worker
- Comprehensive logging with colored output
- Error handling with the option to continue despite failures
- Progress tracking with JSON markers
- Summary reporting for processed counties

### Output Organization
- **GeoParquet Data**: Only GeoParquet files are stored in the output and counties directories
- **Auxiliary Files**: All logs, summary reports, and progress markers are stored in the logs directory:
  - `logs/`: Contains process logs
  - `logs/reports/`: Contains CSV summary reports for each state
  - `logs/progress/`: Contains JSON progress markers for each county

This separation ensures clean data directories while keeping all reporting and logging information well-organized and accessible.

## Summary Reports

The workflow generates detailed summary reports that track the progress and effectiveness of the overlap correction process:

### Metrics Included in Reports
- **Processing Status**: Success, skipped, or failed for each county
- **Feature Counts**: Number of parcels processed in each county
- **Overlap Detection**: Count of overlapping geometries identified and fixed
- **Area Analysis**:
  - Census county area (from TIGER/Line shapefiles)
  - Total parcel area before and after overlap fixes
  - Area discrepancy (difference between census area and parcel area)
  - Percentage of area overlaps removed
  - Improvement in area discrepancy after fixing overlaps

### Summary Statistics
The reports also include aggregated statistics:
- Average area discrepancy across counties (before and after fixes)
- Total overlap area removed
- Counties with the largest remaining discrepancies
- Counties with the most overlaps fixed

These detailed metrics help evaluate the effectiveness of the overlap correction process and identify counties that may need manual review or additional processing.

## Coordinate Reference System (CRS) Handling

The workflow implements a systematic approach to managing coordinate reference systems:

### CRS Transformation Process
1. **Preservation of Source CRS**:
   - During initial GDB to GeoParquet conversion, the original CRS from source geodatabases is preserved
   - The CRS information is logged but not transformed at this stage

2. **Reprojection for Analysis**:
   - In county processing, data is reprojected to EPSG:5070 (USA Contiguous Albers Equal Area projection)
   - This projection is chosen specifically for accurate area calculations
   - Original CRS is stored for reference

3. **Area Calculations in Projected CRS**:
   - All geometric operations (overlap detection, area calculations) are performed in the projected CRS
   - Area values are stored in both square meters and acres while in the appropriate projection

4. **Standardization for Output**:
   - Final data is reprojected to EPSG:4326 (WGS84) for compatibility with BigQuery and other systems
   - Geometry validity is verified after reprojection, with automatic repairs as needed

This CRS management ensures:
- Geometric accuracy for area calculations and overlap detection
- Consistent spatial operations across different states and regions
- Compatibility with downstream database systems and GIS applications
- Proper documentation of all coordinate system transformations
